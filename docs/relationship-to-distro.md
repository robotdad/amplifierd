# How amplifierd Relates to distro-server

This document explains the relationship between amplifierd and distro-server,
two components in the amplifier-distro ecosystem that serve different purposes
but overlap in some areas today. It is written honestly about the current state
while describing where the architecture is heading.

## Two Servers, Two Purposes

| | distro-server | amplifierd |
|---|---|---|
| **Role** | Experience server | Capability server |
| **Purpose** | End-user surfaces: chat UI, Slack, voice | Raw Amplifier primitives over HTTP |
| **Audience** | End users (via surfaces) | Programmatic consumers: CLIs, editors, CI, services, generated apps |
| **Concerns** | Authentication, multi-user, rendering, surface-specific protocols | Sessions, bundles, events, modules, configuration |
| **Abstraction level** | High -- optimized for specific interaction patterns | Low -- exposes primitives that clients compose |
| **Analogy** | A web app | An API server |

**distro-server** is built for people who interact with Amplifier through a
product surface. It handles the concerns those surfaces need: user accounts,
rendering chat messages, managing Slack threads, processing voice input. It
translates between surface-specific protocols and the underlying Amplifier
capabilities.

**amplifierd** is built for machines and developers. It exposes Amplifier's core
primitives -- session lifecycle, bundle management, event streaming, module
invocation -- as a straightforward HTTP API. Any client that can make HTTP
requests can use it: a CLI, an editor extension, a CI pipeline, another service,
or an app generated by an AI agent.

The two are complementary, not competing. distro-server is (or will be) a
consumer of amplifierd, the same way a CLI or editor extension is.

## Current State

There is real duplication between the two servers today. Both implement:

- Configuration management endpoints (reading and writing bundle/config state)
- Session lifecycle management (creating, resuming, listing sessions)
- Some overlapping bundle introspection logic

This duplication exists because the two servers developed in parallel before the
architectural boundaries were fully drawn. It creates maintenance burden and
risks divergent behavior -- a config change made through distro-server might not
be visible through amplifierd, or vice versa.

This is being actively resolved.

## Intended Architecture

The target architecture has a clear layering:

```
  Clients
  -----------------------------------------------
  Chat UI    Slack    Voice    CLI    Editors    CI
     |         |        |       |       |        |
     v         v        v       |       |        |
  +-------------------------+   |       |        |
  |      distro-server      |   |       |        |
  |   (experience layer)    |   |       |        |
  |  auth, rendering, UX    |   |       |        |
  +-------------------------+   |       |        |
              |                 |       |        |
              v                 v       v        v
  +------------------------------------------------+
  |               amplifierd                        |
  |           (capability layer)                    |
  |  sessions, bundles, events, modules, config     |
  +------------------------------------------------+
              |                 |
              v                 v
  +---------------------+  +------------------------+
  |   amplifier-core    |  | amplifier-foundation   |
  |  (runtime engine)   |  |  (shared primitives)   |
  +---------------------+  +------------------------+
```

In this model:

- **Experience clients** (chat UI, Slack, voice) go through distro-server, which
  adds authentication, multi-user management, rendering, and surface-specific
  protocol handling before calling amplifierd.
- **Programmatic clients** (CLI, editors, CI pipelines) call amplifierd directly.
  They don't need the experience layer.
- **amplifierd** is the single source of truth for all Amplifier primitives. No
  other component implements its own session management, config storage, or
  bundle lifecycle.
- **amplifier-core** and **amplifier-foundation** provide the underlying runtime
  and shared types that amplifierd builds on.

## Migration Path

The migration from the current duplicated state to the intended architecture
involves several steps:

1. **Configuration endpoints move to amplifierd.** The daemon owns configuration
   introspection and mutation. distro-server stops implementing its own config
   endpoints and instead calls the daemon's. This is the highest-priority
   migration because config divergence causes the most user-visible confusion.

2. **Session management consolidates.** amplifierd becomes the sole manager of
   session lifecycle. distro-server delegates session creation, resumption, and
   listing to the daemon, adding its own experience-layer concerns (e.g., which
   user owns which session) on top.

3. **Bundle lifecycle moves to amplifierd.** Bundle loading, validation, and
   hot-reload are daemon responsibilities. distro-server consumes bundle state
   from the daemon rather than managing it independently.

4. **distro-server becomes a pure experience layer.** Once the delegation is
   complete, distro-server's responsibilities are cleanly scoped: authentication,
   authorization, multi-user, rendering, and surface-specific protocols. All
   Amplifier capability concerns live in amplifierd.

This migration is incremental. Each step reduces duplication and can be done
independently. The goal is not a big-bang rewrite but a steady transfer of
responsibility to the correct layer.
